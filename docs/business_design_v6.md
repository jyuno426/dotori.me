# 도토리 (DOTORI)

**자산배분 ETF 투자자를 위한 포트폴리오 트래커 + 리밸런싱 계산기 + 은퇴 플래너**

2026년 4월 | dotori.me | v6.0 | Confidential

---

## 1. Executive Summary

**도토리**는 한국의 자산배분 ETF 투자자를 위한 올인원 서비스로, 다음 세 가지 핵심 모듈을 하나로 통합한다:

| 모듈 | 역할 | 기존 대안 (파편화) |
|------|------|-------------------|
| 포트폴리오 트래커 | 다계좌 통합 보유 현황, 비중 분석, 수익률 역산 | 엑셀 / 더리치 / 도미노 |
| 리밸런싱 계산기 | 드리프트 감지, 매수·매도 수량 계산, 월 적립 배분 | etf-rebalancing.com / 엑셀 |
| 은퇴 플래너 | 목표 자산 시뮬레이션, 인출 시뮬레이션 | 삼성증권 계산기 / 스프레드시트 |

**시장 공백:** 한국에서 절세계좌(ISA, IRP, 연금저축)의 과세 특성을 반영하고, 다계좌를 1 포트폴리오로 통합하며, 증권사 API 없이 작동하는 서비스는 존재하지 않는다.

**도메인:** dotori.me | **태그라인:** "한 알씩 모아, 단단한 내일로"

## 2. 타겟 사용자와 비전

도토리는 단순한 도구가 아니라, 자산배분 투자라는 **프레임워크 자체를 제공**하는 서비스이다. 이미 자산배분을 실천하는 숙련 투자자뿐 아니라, 처음 투자를 시작하는 사람까지 같은 프레임워크 안에서 합리적이고 과학적이며 본인이 관리 가능한 투자를 실현하게 하는 것이 비전이다.

### 2.1 사용자 세그먼트

| 세그먼트 | 특성 | 도토리의 역할 |
|---------|------|-------------|
| 숙련 자산배분 투자자 | 이미 ETF 자산배분을 실천 중. 엑셀로 관리하며 리밸런싱 주기를 지키려 노력. 절세계좌 활용 중. 보유 종목 5~10개. | 파편화된 도구를 통합. 리밸런싱·월 적립·배당 추적의 마찰을 제거. 세금 효율적 라우팅 제공. |
| 은퇴설계 투자자 | 은퇴를 목표로 장기 적립 투자 중. 글라이드 패스 개념을 이해하거나 관심 있음. 부부 계좌 통합 관리 필요. | 다계좌 통합 관리 + 은퇴 시뮬레이션. 장기 목표 대비 현재 진행 상황을 시각화. |
| 투자 입문자 | 투자해야 한다는 건 아는데 어떻게 시작할지 모름. 정보 과잉 속 결정 장애. 개별 종목 투자의 리스크는 직감적으로 인지. | 도토리가 제안하는 프레임워크(자산배분 + ETF + 월 적립 + 리밸런싱) 안에서 첫 투자를 시작. "올바른 방법"이 "가장 쉬운 방법"이 되도록 설계. |

### 2.2 입문자를 위한 프레임워크 제공

투자 입문자에게 가장 큰 장벽은 지식이 아니라 의사결정이다. "어떤 종목을 살까" "얼마나 살까" "언제 팔까" — 이 모든 질문에 매번 답을 내려야 하는 구조가 투자를 시작하지 못하게 만든다. 도토리는 이 의사결정 부담을 프레임워크로 흡수한다:

- **"어떤 종목을"** → 자산군별 ETF 조합 (주식/채권/대안자산 비중만 결정하면 됨)
- **"얼마나"** → 목표 비중에 따라 시스템이 자동 계산
- **"언제"** → 매월 적립일에 맞춰 시스템이 매수 수량 제안
- **"언제 팔까"** → 리밸런싱 시점에만 시스템이 안내, 그 외에는 보유 유지

입문자는 처음에 "주식 60% / 채권 40%" 같은 단순한 목표 비중 하나만 설정하면, 이후의 모든 매수·리밸런싱·배당 처리를 도토리의 가이드 안에서 수행할 수 있다. 복잡한 이론을 먼저 공부할 필요 없이, 사용하면서 자연스럽게 자산배분 투자를 체득하게 되는 구조이다.

### 2.3 세그먼트별 진입 경로

| 세그먼트 | 온보딩 | 첫 번째 가치 경험 |
|---------|--------|-----------------|
| 숙련 투자자 | 기존 보유 종목·수량 수기 입력 (5분) | 흩어진 계좌를 한눈에 보는 통합 비중 분석 |
| 은퇴설계 투자자 | 계좌 등록 + 은퇴 목표 설정 | 현재 자산으로 은퇴 목표까지 얼마나 남았는지 시각화 |
| 투자 입문자 | 목표 비중 설정 (주식/채권 비율만 선택) | 첫 월 적립금으로 매수할 ETF와 수량을 바로 안내 |

세 세그먼트 모두 같은 제품을 사용하되, 진입 시점의 가이드와 첫 경험이 다르다. 입문자는 사용하면서 숙련 투자자가 되고, 숙련 투자자는 은퇴설계로 확장한다. 도토리 안에서 투자자의 성장 경로가 자연스럽게 이어지는 구조이다.

## 3. 해결하려는 문제

### 3.1 도구의 파편화 — 그리고 도토리의 해결 방식

현재 자산배분 ETF 투자자는 하나의 워크플로우에 5~6개 도구를 왔다갔다한다. 도토리는 이 전체 과정을 하나의 서비스 안에서 끊김 없이 처리한다:

| 단계 | 현재 (파편화) | 도토리에서 |
|------|-------------|----------|
| 전략 수립 | Portfolio Visualizer / testfol.io | 목표 비중 설정 (자산군별 % 입력) |
| 보유 현황 기록 | 엑셀 수기 입력 | 수기 입력 (5~10개 종목, 5분) + 배당 자동 입력 |
| 비중 모니터링 | 엑셀 수식 관리 | 실시간 비중 분석 + 드리프트 자동 감지 |
| 리밸런싱 계산 | etf-rebalancing.com / 엑셀 | ETF 1주 단위 매수·매도 수량 자동 계산 |
| 매매 실행 | 증권사 HTS/MTS | 도토리가 종목별 매매 지시서를 생성 → 사용자가 MTS에서 실행 |
| 매매 후 반영 | 엑셀에 수동 기록 | "완료" 버튼으로 일괄 반영 |
| 배당/분배금 추적 | ETFCheck / 증권사 앱 확인 | 공공 API 자동 감지 → 예수금 증가 레코드 자동 생성 |
| 은퇴 시뮬레이션 | 별도 계산기 / 스프레드시트 | 실제 포트폴리오 데이터 기반 시뮬레이션 |

### 3.2 리밸런싱의 마찰

리밸런싱 수량을 계산하고, 증권사 앱에서 직접 매매하고, 다시 엑셀에 기록하는 과정이 번거로워서 결국 리밸런싱을 미루게 된다. Vanguard 연구에 따르면, 리밸런싱을 하지 않은 포트폴리오는 10년 후 위험(표준편차)이 원래 설계보다 30~50% 높아진다.

### 3.3 다계좌 관리의 복잡성

한국 투자자는 절세계좌 3총사(ISA, IRP, 연금저축) + 일반 위탁 + 배우자 계좌까지 4~8개 계좌를 운용하는 경우가 흔하다. 기존 서비스들은 계좌별로 보여주지만, 자산배분 투자자에게 필요한 것은 전체 포트폴리오 단위의 비중 분석이다.

## 4. 솔루션: 도토리의 핵심 기능

### 4.1 포트폴리오 트래커

다계좌 통합 보유 현황 관리와 투자 성과 분석의 기반이 되는 핵심 모듈이다.

#### 다계좌 통합 구조

계좌(Account)와 포트폴리오(Portfolio)를 분리하여, 복수 계좌를 1개 포트폴리오로 묶어 관리한다. 각 계좌에는 세금 유형(과세이연/비과세/분리과세/일반과세) 속성만 붙인다.

- 지원 계좌 유형: IRP, 연금저축, ISA, 일반 위탁, CMA
- 배우자/가족 계좌 통합 지원
- 전체 포트폴리오 기준 자산군별·종목별 비중 분석
- 드리프트(목표 비중 대비 현재 비중 괴리) 자동 감지 및 알림

#### 투자 성과 지표

사용자의 투자 성과를 다양한 관점에서 정확하게 측정하는 수익률 지표를 제공한다:

| 지표 | 설명 | 용도 |
|------|------|------|
| 시간가중수익률 (TWR) | 입출금 영향을 제거한 순수 투자 성과 | 벤치마크(코스피, S&P500 등) 대비 비교 |
| 금액가중수익률 (MWR) | 실제 입출금 타이밍을 반영한 개인 수익률 | "내가 실제로 얼마 벌었나" |
| 원금 대비 수익률 | 총 투자원금 대비 현재 평가액의 단순 비율 | 직관적 이해, 초보자 친화적 |
| 배당 수익률 | 보유 자산 대비 연간 배당/분배금 비율 | 인컴 투자자를 위한 현금흐름 파악 |
| 세후 수익률 (추정) | 계좌 유형별 과세 특성을 반영한 추정 수익률 | 절세계좌 효과 시각화 (세법 리서치 후 구현) |

- 기간별 조회: 1개월, 3개월, YTD, 1년, 3년, 전체 기간
- 벤치마크 비교: 코스피, S&P500, 60/40 포트폴리오 등 사용자 설정 가능
- 자산군별 기여도 분해: 주식/채권/대안자산 각각이 전체 수익에 기여한 정도

#### 데이터 입력 방식

- 수기 입력: 종목별 보유 수량과 예수금 잔액을 직접 입력 (입출금 발생 시 금액도 입력)
- 배당 자동 입력: 공공 API 감지 시 예수금 증가 레코드 자동 생성 → 사용자 입력 부담 감소
- 월 적립 반자동 반영: 적립 실행 후 "완료" 시 보유 수량 + 예수금 일괄 업데이트
- 언제든 오버라이드: 현재 보유 수량이나 예수금이 맞지 않으면 직접 수정 가능
- CSV 임포트: 검토 중 (아래 비고 참조)

> **CSV 임포트에 대한 비고:** 증권사 HTS에서 거래내역 엑셀을 다운로드하여 업로드하는 방식은 기술적으로는 구현 가능하지만, 사용자 관점에서 "HTS 열기 → 조회 → 다운로드 → 파일 찾기 → 업로드"의 과정이 수기로 5~10개 종목을 입력하는 것보다 실제로 쉬운지는 검증이 필요하다. 자산배분 ETF 투자자의 보유 종목 수는 대체로 5~10개로 적기 때문에 초기 셋업도 수기 입력으로 5분이면 충분하다. 과거 거래 이력이 긴 사용자의 수익률 복원 용도로는 유효할 수 있으므로, MVP에서는 제외하고 사용자 피드백으로 필요성이 검증되면 추가한다.

#### 배당/분배금 자동 감지 — 입력 대행

공공데이터포털 주식배당정보 API + SEIBro ETF 분배금 API + DART OpenAPI를 조합하여 보유 종목의 배당/분배금 발생을 자동 감지한다. 감지 시 예수금 증가 레코드를 자동 생성하여 **사용자의 수기 입력을 대행**한다. 자동 생성된 레코드가 실제와 다르면, 사용자가 예수금 잔액을 오버라이드하면 된다. 감지에 실패하더라도 사용자가 직접 예수금을 업데이트하면 수익률 계산에 문제가 없다.

### 4.2 리밸런싱 계산기

#### 리밸런싱 라우팅 — 최적 계좌에서만 매매

리밸런싱이 필요할 때, 시스템이 세금 비용이 가장 낮은 계좌 1곳을 자동 선택하여 해당 계좌에서만 매매하도록 제안한다.

| 우선순위 | 계좌 유형 | 매매 시 세금 효과 | 적합 상황 |
|---------|----------|-----------------|---------|
| 1순위 | IRP / 연금저축 | 과세이연 (매매 시점 세금 없음) | 모든 리밸런싱 |
| 2순위 | ISA | 비과세 한도 내 유리 | 1순위 부족 시 |
| 3순위 | 일반 위탁 | 양도세 등 발생 가능 | 최후 수단 |

#### 월 적립 캐시플로우 리밸런싱

매월 적립금 투입 시 부족 비중 자산에 우선 배분하여 매도 없이 비중을 교정한다. 매도가 발생하지 않으므로 세금 부담이 최소화되며, 사용자의 가장 빈번한 행동(월 적립)에 자연스럽게 녹아든다.

| 모드 | 조건 | 동작 |
|------|------|------|
| 균등 배분 | 드리프트 ±3% 이내 | 목표 비중대로 분배 매수 |
| 부족 우선 | 특정 자산 목표 대비 5%+ 부족 | 적립금 전액 부족 자산에 집중 |
| 강제 리밸런싱 | 드리프트 10%+ 초과 | 매도 포함 리밸런싱 안내 |

**세액공제 한도 기반 계좌 배분:** 적립금을 IRP(연 700만 원) → 연금저축(연 600만 원) → ISA → 일반 순으로 자동 배분하여 세액공제 극대화와 비중 교정을 동시에 처리한다.

#### HTS/MTS 연결: 매매 지시서 패턴

도토리는 증권사 API를 사용하지 않으므로 직접 주문을 넣을 수 없다. 대신 사용자가 증권사 MTS에서 매매를 실행할 때 최소한의 마찰로 처리할 수 있도록 다음과 같은 **매매 지시서** 패턴을 사용한다:

| 단계 | 도토리 화면 | 사용자 행동 | 소요 시간 |
|------|-----------|-----------|---------|
| 1. 계산 | 종목별 매수·매도 수량 리스트 생성 | 확인만 | 자동 |
| 2. 지시서 확인 | 종목명, 수량, 현재가를 한눈에 정리 | MTS로 전환하며 참조 | 10초 |
| 3. MTS 매매 | — | MTS에서 종목별로 주문 실행 | 3~5분 |
| 4. 완료 반영 | "매매 완료" 버튼 표시 | 버튼 클릭 → 보유량 일괄 업데이트 | 5초 |
| 5. 예수금 보정 | 예수금 입력 필드 표시 | MTS에서 확인한 예수금 입력 | 10초 |

MTS 매매를 포함한 전체 플로우는 약 **5분** 내외이며, 기존 방식(엑셀 계산 → MTS 매매 → 엑셀 기록)의 20~30분 대비 대폭 단축된다. 핵심은 "계산과 기록"을 시스템이 처리하고, 사용자는 "매매 실행"만 직접 하는 분업 구조이다.

**향후 확장:** 한국투자증권 KIS Developers 등 API를 제공하는 증권사가 늘어나면, 해당 증권사 사용자에 한해 도토리 내 원클릭 주문 기능을 추가할 수 있다. 다만 MVP에서는 API 의존 없는 범용 구조를 우선한다.

### 4.3 은퇴 플래너

도토리의 은퇴 플래너는 실제 포트폴리오 데이터와 연동된 시뮬레이션을 제공한다. 별도의 계산기가 아니라, 현재 보유 자산·월 적립액·수익률 이력이 자동으로 반영되는 것이 핵심이다.

#### MVP 범위

- 목표 자산 도달 시뮬레이션: 월 적립액과 예상 수익률로 목표 금액 도달 시점 예측
- 인출 시뮬레이션: 은퇴 후 월 인출액 설정 시 자산 유지 기간 계산
- 인플레이션 반영: 물가상승률 적용한 실질 구매력 기준 시뮬레이션

#### 향후 고도화 (Post-MVP)

MVP 이후 단계적으로 시뮬레이션 엔진을 고도화한다. 몬테카를로 시뮬레이션 기반의 확률적 은퇴 성공률 분석, 사용자 맞춤형 글라이드 패스 설계 도구, 국민연금·퇴직연금 수령액을 포함한 종합 은퇴 소득 분석, 세금 최적화를 반영한 계좌별 인출 순서 제안 등을 검증된 사용자 피드백을 기반으로 우선순위를 정해 추가할 예정이다.

### 6.2 데이터 모델 — 3개 데이터, 역산 방식

데이터 모델은 의도적으로 단순하게 설계한다. 단가·수수료·세금·배당금 등을 개별 추적하지 않고, 각 일자별로 3개 데이터만 기록하면 시스템이 모든 수익률 지표를 도출할 수 있다.

#### 일자별 기록 데이터: 3개

| 데이터 | 입력 주체 | 설명 |
|--------|---------|------|
| 입출금 금액 | 사용자 | 해당 일자에 외부에서 들어오거나 나간 금액 (월급 입금, 출금 등) |
| 종목별 보유 수량 | 사용자 | 각 종목의 변경 후 총 보유 수량 (매매 발생 시 업데이트) |
| 예수금 잔액 | 사용자 | 해당 일자 기준 예수금 잔고 (MTS에서 확인 후 입력) |

#### 자동 수집 데이터

| 데이터 | 수집 방식 | 용도 |
|--------|---------|------|
| 종목별 종가 | cronjob (매일 장 마감 후) | 포트폴리오 평가액 계산, 수익률 도출 |
| 배당/분배금 공시 | 공공 API 폴링 | 감지 시 예수금 증가 레코드 자동 생성 (입력 대행) |

#### 수익률 역산 원리

수익률 계산의 핵심은, 예수금 변화 중 입출금으로 설명되지 않는 부분을 투자 수익으로 역산하는 것이다:

> **투자 수익 = 예수금 변화량 - 입출금 합계**

이 역산 금액에는 배당금, 분배금, 매매 수수료, 세금 등이 모두 순(net)으로 합산되어 있다. 개별 항목을 분리할 필요가 없다. 사용자가 입력하는 예수금 잔액은 이미 증권사가 수수료와 세금을 정산한 후의 금액이기 때문이다.

**예시:**

- 전일 예수금 100만 원 → 오늘 월급 50만 원 입금 → 예수금 잔액 155만 원 입력
- 시스템 역산: 155 - 100 - 50 = **5만 원이 투자 수익** (배당/분배금 등)
- 공공 API가 "삼성전자 배당 5만 원"을 감지했다면, 해당 금액으로 레코드를 자동 생성
- API 감지가 실패해도 수익률 계산에 영향 없음 — 역산으로 이미 반영됨

#### 설계 원칙

- **최소 완전 데이터셋:** 3개 데이터 + 종가(자동 수집)만으로 TWR, MWR, 원금 대비 수익률 등 모든 지표를 계산할 수 있다. 이보다 적으면 부족하고, 더 많으면 불필요하다.
- **스냅샷 기반:** 보유 수량과 예수금은 "변경 후 절대값"을 기록한다. 언제든 사용자가 현재 상태를 직접 입력하여 보정할 수 있다.
- **단가·수수료·세금·배당금 미기록:** 이 값들은 증권사마다 계산 방식이 다르고, 사용자 입력 시 오류 가능성이 높다. 예수금 잔액에 이미 모두 반영되어 있으므로 별도 기록이 불필요하다.
- **배당 감지는 입력 대행:** 공공 API가 배당/분배금을 감지하면 예수금 증가 레코드를 자동 생성한다. 사용자가 직접 입력하는 수고를 줄여주는 것이 목적이며, 자동 생성 값이 부정확하면 사용자가 예수금 잔액을 오버라이드하면 된다. 감지 실패 시에도 수익률 계산에 영향 없다.

### 6.3 한국 세법 반영 방향

절세계좌별 과세 특성(과세이연, 비과세 한도, 분리과세 등)은 서비스의 핵심 차별점이지만, 구체적인 세율과 계산 로직은 구현 단계에서 현행 세법을 기반으로 정밀하게 리서치한다. 세법은 매년 개정될 수 있으므로, 세금 로직을 모듈화하여 변경에 빠르게 대응하는 구조를 지향한다.

구현 시 리서치가 필요한 주요 영역:

- 계좌 유형별 매매차익 과세 방식 (국내 ETF, 해외 ETF, 채권형 ETF 등)
- 배당소득세 원천징수 및 세후 실수령액 계산 로직
- 금융소득종합과세 기준과 알림 조건
- IRP/연금저축 내 과세이연 효과 및 수령 시 세율
- ISA 비과세 한도 및 초과분 분리과세 규정
- 세액공제 한도(IRP, 연금저축) 연간 상한 및 적용 조건

MVP에서는 계좌 유형별 과세 특성을 "과세이연 / 비과세 한도 / 일반과세" 수준으로 단순 분류하여 리밸런싱 라우팅의 우선순위에 활용하고, 정밀한 세후 수익률 계산은 Post-MVP에서 단계적으로 고도화한다.

## 7. 기술 스택 및 개발 로드맵

### 7.1 저비용 기술 스택

사용자가 증가해도 인프라 비용이 선형적으로 증가하지 않는 구조를 지향한다. 서버리스와 엣지 컴퓨팅을 활용하여 트래픽에 따라 자동 스케일링하되, 유휴 시 비용이 0에 수렴하는 아키텍처를 설계한다.

| 계층 | 기술 | 비용 구조 | 선택 근거 |
|------|------|---------|---------|
| Frontend | Next.js + Tailwind (PWA) | Cloudflare Pages (무료~$5/월) | 정적 배포, CDN 자동, 모바일 앱 대체 |
| Backend API | Hono / Next.js API Routes | Cloudflare Workers (무료~$5/월) | 요청당 과금, 유휴 비용 0원 |
| Database | Turso (libSQL) 또는 Supabase | 무료~$10/월 (5GB) | SQLite 호환, 엣지 복제, 저비용 확장 |
| 시세 캐시 | Cloudflare KV / R2 | 무료 티어 → $0.5/백만 읽기 | Redis 대비 10배 저렴, 글로벌 엣지 |
| 배치 작업 | Cloudflare Cron Triggers | Workers 요금에 포함 | 별도 스케줄러 서버 불필요 |
| 파일 저장 | Cloudflare R2 (S3 호환) | $0.015/GB/월, 이그레스 무료 | CSV 업로드, 백업 |

#### 비용 시뮬레이션

| 규모 | MAU | 예상 월 인프라 비용 | 사용자당 비용 |
|------|-----|-------------------|-------------|
| 초기 (베타) | ~500 | 무료 (프리 티어) | 0원 |
| Y1 | 1,500 | ~$15 (약 2만 원) | ~13원 |
| Y2 | 8,000 | ~$50 (약 7만 원) | ~9원 |
| Y3 | 30,000 | ~$150 (약 20만 원) | ~7원 |

> Railway/Fly.io + Redis + PostgreSQL 조합 대비 Cloudflare 풀스택이 동일 MAU 기준 약 5~10배 저렴하다. 핵심은 상시 가동 서버가 아닌 요청 기반 과금 구조이며, 시세 데이터 캐싱에 KV를 사용하면 외부 Redis 서비스가 불필요하다.

| 확인 | 앱 접속 시 | 자동 생성된 레코드 표시 | 맞으면 넘어감 |
| 보정 | 금액 불일치 시 | — | 예수금 잔액 오버라이드 |

> 감지 실패 시에도 사용자가 예수금 잔액만 정확히 입력하면 수익률 계산에 문제가 없다. 자동 입력은 사용자의 수기 입력 빈도를 줄여주는 편의 기능이다.

## 11. 브랜드 아이덴티티

### 11.1 브랜드 스토리

다람쥐는 겨울을 대비해 도토리를 부지런히 모은다. 한 번에 많이 모으는 것이 아니라, 매일 조금씩, 차곡차곡. 도토리는 이런 다람쥐의 지혜를 따르는 투자자를 위한 서비스이다. 매월 꾸준히 적립하고, 균형을 맞추고, 겨울(은퇴)이 왔을 때 풍요롭게 보내는 것. 도토리는 그 여정의 조력자이다.

### 11.2 이름·도메인·태그라인

| 항목 | 내용 |
|------|------|
| 서비스명 | 도토리 (Dotori) |
| 도메인 | dotori.me (연 8,800원) |
| 태그라인 (한국어) | "한 알씩 모아, 단단한 내일로" |
| 태그라인 (영어) | "One acorn at a time, toward a solid tomorrow" |

### 11.3 마스코트 & 톤

**마스코트:** 도토리를 양 볼에 물고 있는 다람쥐 — 귀엽고 친근하면서도 부지런한 이미지.

**톤:** 친근하고 신뢰감 있는 조력자. 복잡한 금융 용어를 쉽게 풀어주고, 사용자가 스스로 판단할 수 있도록 도와주는 톤. 금융 자문이 아닌 도구로서의 정체성을 명확히 한다.

## 12. 요약

도토리는 한국 자산배분 ETF 투자자를 위한 유일한 올인원 서비스로, 포트폴리오 트래커·리밸런싱 계산기·은퇴 플래너 세 가지 모듈을 하나로 통합한다. 숙련 투자자의 도구 파편화를 해결하는 것을 넘어, 투자 입문자가 도토리의 프레임워크 안에서 합리적이고 과학적이며 스스로 관리 가능한 투자를 처음부터 실현할 수 있게 하는 것이 비전이다.

**핵심 차별화:**

- 3-in-1 통합: 트래커 + 리밸런싱 + 은퇴 플래너를 한 서비스에서
- 다계좌 통합 + 리밸런싱 라우팅 (한국 절세계좌 전용)
- 월 적립 캐시플로우 리밸런싱 (매도 없는 비중 교정)
- 매매 지시서 패턴: 계산·기록은 시스템, 매매 실행만 사용자 (MTS 5분)
- 다양한 수익률 지표: TWR, MWR, 원금 대비, 배당 수익률
- 증권사 API 미사용 — 법적 리스크 제로
- 최소 완전 데이터 모델: 입출금 + 보유수량 + 예수금 잔액 3개만 기록, 수익률은 역산
- 배당/수수료/세금 개별 추적 불필요 — 예수금 잔액에 이미 반영, 배당 감지는 자동 입력 대행
- Cloudflare 풀스택으로 MAU 3만에서도 월 20만 원 이하 인프라 비용
- 입문자부터 숙련자까지 — 프레임워크가 곧 제품
